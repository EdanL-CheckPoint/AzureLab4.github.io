Topology

Objective

This lab will simulate an issue with a customer's environment.

  • Following some cloud maintenance on the customer side, the customer can no longer reach their web server
  • The customer hasn't had a chance to check anything yet

Lab objectives

  • Connecting the the Web Browser uses the frontend load balancer IP (frontend-lb)
  • Connecting to the web browser with HTTP will show a simple LAMP page
  • Note: The connection is only HTTP. HTTPS will not work
  • Goal:
    • Find the root cause of the issue
    • Resolve the issue and manage to connect via HTTP to the web server

Post Deployment Actions

Step 1

Make sure the MGMT server has finished FTW.

  • Connect with SSH to the MGMT server
  • If you are in expert mode, the machine is done with FTW. If not, wait until the VM finishes FTW and automatically reboots again.
  • Check that the MGMT is completely up by running api status | Do not proceed forward before this is ready!

Step 2

Run the Auto-MGMT script.

  • Note: In SmartConsole this lab will be called AzureLab1, this is correct and you can ignore the naming scheme
  • Connect with SSH to the MGMT server
  • Connect to the Azure portal using the credential proivded in the lab environment tab
  • Run the following command on the MGMT server: /var/log/automgmt.sh
  • Following the script prompt, fill in the Gaia version "R82.10", the public IPs of the cluster members as they are written in Azue portal and the private IP of the web server
  • Once the script is done, connect to SmartConsole and open the cluster properties
  • Open the network topology tab for eth1 and change the topology to "Override"
  • Push policy

  • Note: Changing the topology to "override" is not necessary a real world scenario. This step is there to fix a lab automation issue.
