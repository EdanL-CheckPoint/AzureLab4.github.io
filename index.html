Topology

Objective

This lab will be split to 3 diferent problems which you will need to solve in the assigned order

  • Following some critical failures in the customer's environment, thye have redeployed a new managment server and have redeployed thier VMSS solution via terraform
  • The customer is a direct customer and has different issues following the redeployment

Lab 4 - Problem A

Lab objectives

  • After configuring the CME, the VMSS instance are not showing on SmartConsole
  • All the CME details input in SmartConsole were checked and are correct
  • Goal:
    • Find the root cause of the issue
    • Resolve the issue and have the gateways appear in SmartConsole

Lab 4 - Problem B

Lab objectives

  • Testing scale-out is not working.
  • Simulatiing high CPU on the gateways does not create a new instance
  • Goal:
    • Simulate high CPU on by running the following command on both gateways 4 times (1 for each core of the VM): yes > /dev/null &
    • Resolve the issue and have the gateways deploy a third instance due to high CPU utilization

    • Note: to stop the hgih CPU utilization run #killall yes

Lab 4 - Problem C

Lab objectives

  • Customer has issue which requires adding a custom parameter to fwkern.conf
  • This means the customer needs this parameter to be present when a new instance appears during scale out
  • Goal:
    • Find a method to add the following parameter to fwken.conf when a new instance scales out using CME: Lab4=True
    • Test your soultion by scaling out and seeing if the parameter is there after deployment


Post Deployment Actions

Step 1

Make sure the MGMT server has finished FTW.

  • Connect with SSH to the MGMT server
  • If you are in expert mode, the machine is done with FTW. If not, wait until the VM finishes FTW and automatically reboots again.
  • Check that the MGMT is completely up by running api status | Do not proceed forward before this is ready!

Step 2

Run the Auto-MGMT script.

  • Connect with SSH to the MGMT server
  • Run the following command on the MGMT server: /var/log/automgmt.sh
  • Once the script is done, connect to SmartConsole

Step 3

Conect the Azure Cloud account to CME

  • In SmartConsole navigate to Settings > CloudGuard Network
  • Click on Advanced in the top right corner and change the Management name to Lab-mgmt
  • Add an Azure account and fill in the detaails of the account service principal located in the Lab environment page

Step 4

Create a template for the VMSS environment

  • In the Azure section where you connected the account, clikc on the new button of Add template
  • Name the template Lab-tn choose the correct version and add the SIC password Checkpoint123
  • Note: R82.10 might be missing from dropdown option, choose another option for now, after the gateways will be scanned by the CME, the version R82.10 will appear
  • Connect with SSH to the management server and run tail -f /var/log/CPcme/cme.log
  • Once you see the CME logs showing the gateways, the lab is ready
